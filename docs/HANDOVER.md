# Передача проекта клиенту — ОКНА CRM (Ocean Glass)

Документ описывает всё, что нужно сделать, чтобы полностью передать CRM клиенту
в боевую эксплуатацию.

Обозначения: ✅ готово · 🔴 блокер · 🟡 важно · ⚪ опционально ·
👤 нужны данные/решение клиента · 🤖 может сделать исполнитель (Pllato).

---

## 1. Что это и где работает

- **Продукт:** CRM для оконного/стекольного производства (воронка, замер+КП,
  склад профиля/стеклопакетов, производство, финансы, права доступа).
- **Стек:** статичный фронтенд (vanilla HTML/CSS/JS, без сборки) + бэкенд на
  **Cloudflare Pages Functions** + **D1** (база) + **R2** (файлы) + **JWT** (вход).
- **Адреса сейчас:**
  - боевой бэкенд: `https://okna-crm.pages.dev`
  - витрина/прод: `https://pllato.kz/app/okna/`
- **Репозиторий:** `pllato/okna-crm` (GitHub). Деплой — Cloudflare Pages.
- **Ресурсы Cloudflare:** D1 `okna-crm-db`, R2 `okna-crm-files`, Pages-проект `okna-crm`.

## 3. Что нужно сделать для полной передачи

### A. Данные и контент
- 🔴👤 **Реквизиты компании** — заменить демо (город «Ош», телефон, адрес) на
  реальные Ocean Glass. Где: `Настройки → Данные компании` (название, юрлицо,
  город, телефон, производство, оборот). Название «Ocean Glass» уже верное.
- 🟡👤 **Каталог и прайс** — внести реальные позиции и цены: профиль (Rehau/KBE/
  Alutech…), стеклопакеты, открывания, доп-опции; начальные остатки склада и
  минимумы. Редактор: раздел `Каталоги` и `Склад`.
- 🟡👤 **Услуги/продукция** — подтвердить перечень (окна, фасады, стекло, душевые,
  перегородки) для корректных опций расчёта.

### B. Инфраструктура и владение (Cloudflare)
Два варианта — выбрать один:

**Вариант 1. Оставить у Pllato + пригласить клиента (проще, рекомендуется,
если поддержку ведёт Pllato).**
- 🤖 Пригласить аккаунт клиента членом в Cloudflare-аккаунт Pllato с нужной ролью.
- Переносить ничего не нужно; обслуживание и деплой остаются у Pllato.

**Вариант 2. Полный перенос на аккаунт клиента (если нужен полный контроль).**
В Cloudflare нет «передачи проекта» между аккаунтами — ресурсы пересоздаются,
данные переносятся. Нужно:
- 👤 Клиент создаёт аккаунт Cloudflare (бесплатного тарифа достаточно:
  Pages + D1 5 ГБ + R2 10 ГБ).
- 👤 Предоставляет **Account ID** и **API-токен** с правами: Pages (Edit),
  D1 (Edit), **R2 (Edit)**, Workers Scripts (Edit).
- 🤖 Шаги миграции (исполнитель, ~30–40 мин):
  1. Создать на аккаунте клиента: D1 `okna-crm-db`, R2 `okna-crm-files`,
     Pages-проект `okna-crm`.
  2. Прописать новый `database_id` в `wrangler.toml` (бинды `DB`/`BUCKET`
     универсальные — код не меняется).
  3. Накатить схему `schema.sql` и залить данные (есть свежий экспорт D1; R2 пуст).
  4. Задать секреты на новом проекте (см. блок D).
  5. Подключить домен клиента (см. блок F), деплой, проверка входа.
  6. Передать владение аккаунтом Cloudflare; репозиторий GitHub — перенести к
     клиенту или оставить у Pllato и деплоить токеном клиента.

### C. Первый директор и сотрудники
- Регистрации в системе нет (аккаунты заводит директор). Поэтому **первый
  директор создаётся напрямую в базе** — уже есть `director@okna.kz`.
- 🔴👤 Превратить его в реального директора клиента: задать **имя, email** и
  **стойкий пароль**. Способ: войти текущим директором →
  `Настройки → Сотрудники` → изменить директора + «Сменить пароль».
  (или 🤖 одним `UPDATE` в базе при переносе).
- 🟡👤 Директор сам добавляет остальных сотрудников (менеджеры, замерщики, цех,
  склад) и выдаёт им пароли — доступ к базе им не нужен.

### D. Безопасность
- 🔴🤖 **Сгенерировать новый `JWT_SECRET`** (стойкая случайная строка) и задать
  секретом проекта. После смены все текущие сессии разлогинятся — это нормально.
- 🔴👤 **Стойкие пароли** для всех аккаунтов (сейчас у всех временный `okna2026`).
- 🟡🤖 Убрать дефолтный пароль `okna2026` из `seed.sql` (используется только при
  поднятии новой базы с нуля).
- ✅ Уже есть: RBAC, ограничение попыток входа, security-заголовки, HTTPS,
  пароли PBKDF2, деньги скрыты от сборщиков/склада.

### E. Интеграции (поэтапно, после базового запуска)
- ⚪👤 **WhatsApp (Green API):** отдельный платный инстанс + реальный номер;
  привязать вебхук. Входящие заявки → автоматически клиент+сделка (готово).
- ⚪👤 **Instagram:** Business-аккаунт + связанная Facebook-страница + сервис-
  форвардер (~$10–20/мес) для исходящих. Входящие DM/заявки → сделки (готово).
  Нужна ссылка на Instagram компании.
- ⚪👤 **Звонки из браузера (SIP):** VM с Asterisk + SIP-trunk у оператора связи +
  2 секрета (`SIP_DOMAIN`, `SIP_ENDPOINT_PASSWORD`). Кнопки и бэкенд готовы,
  при отсутствии настройки деградируют без ошибок.

### F. Домен
- 🟡👤 Решить адрес: поддомен клиента (напр. `crm.oceanglass.kg`) или остаётся
  `pllato.kz/app/okna`. Для своего домена — подключить Custom Domain в Pages и
  вести DNS на Cloudflare.
- 🟡🤖 Если боевым остаётся `pllato.kz/app/okna` — проверить, что путь `/api`
  проксируется на Pages-проект (иначе вход и серверные функции там не работают;
  на `okna-crm.pages.dev` всё работает).

### G. Эксплуатация
- 🟡 **Резервные копии:** показать клиенту экспорт/импорт (`Настройки → Резервная
  копия`); договориться о регулярности. Серверный бэкап D1 — командой экспорта.
- 🟡👤 **Обучение** сотрудников по ролям (замер→КП, воронка, склад, цех, финансы).
- 🟡👤 **Поддержка:** кто сопровождает и обновляет после передачи.

## 4. Что нужно прислать от клиента (сводка)
1. Реквизиты компании: юрлицо, город, адрес, телефон(ы), email.
2. Прайс и каталог: профили/стеклопакеты/опции и цены; остатки склада.
3. Имя + email реального директора (и пожелания по паролям).
4. Решение по Cloudflare: оставляем у Pllato (приглашение) или полный перенос
   (тогда — Account ID + API-токен с правами Pages/D1/R2/Workers).
5. Домен: нужен ли свой (и какой), доступ к DNS.
6. Для интеграций (позже): WhatsApp-номер, Instagram Business + Facebook-страница.

---

## 5. Минимум, чтобы запустить «в бой»
1. A: реальные реквизиты компании + базовый прайс.
2. B: решение по Cloudflare (перенос или приглашение).
3. C: реальный директор (email + пароль).
4. D: новый `JWT_SECRET` + стойкие пароли.
5. F: определить боевой домен (+ проверить `/api`).

Интеграции (WhatsApp/Instagram/звонки) подключаются поэтапно уже после старта —
они опциональны и не блокируют запуск.
